# ADR 0003: 収録曲データの供給方法

- 状態: 採用
- 日付: 2026-06-14

## 決定

収録曲データ(featured ＋ サンプル曲の曲名・作曲者・難易度・旋律)は **`ScoreRepository` 境界の裏で同梱 Dart コードとして供給する**。実装は `lib/src/data/score_data.dart` + `bundled_score_repository.dart`。サーバ・リモート配信は用意しない。ユーザーが作成・編集した曲は別系統(`LibraryStore`。ADR 0001)で永続化する。

## 背景

`docs/product-spec.md` の未確定事項「収録曲データの供給方法(Dart コード内 / アセット JSON)」を決める。前提として、サーバを用意しない(オフライン完結)ことはプロダクト方針で確定している。候補:

| 候補 | 評価 |
| --- | --- |
| 同梱 Dart コード | 型チェックとデータ不変条件の contract test がコンパイル時 + unit test で効く。パース層が不要 |
| アセット JSON | データだけの差し替えが可能だが、パース・スキーマ検証コードとその失敗系テストが必要になる。アプリ更新なしの配信ができるわけではない(どのみち再ビルド) |
| リモート配信 | サーバが必要。プロダクト方針(バックエンドなし)に反する |

## 理由

- アセット JSON の利点「コードとデータの分離」は、配信経路がアプリバイナリしかない本アプリでは実利がない。JSON 化はパース失敗という新しい失敗モードとテスト対象を増やすだけになる(「必要になるまで複雑にしない」)。
- データ不変条件(音名が妥当 / 音価が許容値 / beat 非負・整列可能 / id 一意 / stars 0–5)は `test/unit/data/` の contract test で保証でき、Dart コードのままが最も検証しやすい。
- `ScoreRepository` 境界は維持するため、収録曲が大幅に増える・外部エディタでデータを書くようになる等の要件が出たら、実装ごとアセット JSON に差し替えられる。その時点で新しい ADR を書く。

## 結果

- 収録曲の追加・修正は `score_data.dart` の編集 + contract test の green で完結する。
- テストは `test/fixtures/` の最小 fixture リポジトリも併用し、ロジックテストは本番データに依存させない。
- プロトタイプは全曲が同一のデモ旋律(`DEFAULT`)を共有している。本実装では曲ごとの実旋律へ順次差し替える(差し替えても contract test が不変条件を守る)。
